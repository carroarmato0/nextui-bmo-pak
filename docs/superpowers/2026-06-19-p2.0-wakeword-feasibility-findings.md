# P2.0 — On-Device Wake-Word Feasibility Spike: Findings

**Date:** 2026-06-19
**Spec:** [`specs/2026-06-19-self-hosted-stt-tts-and-wake-word-design.md`](specs/2026-06-19-self-hosted-stt-tts-and-wake-word-design.md) (Phase 2, task P2.0)
**Branch:** `spike/p2.0-wakeword-feasibility`
**Verdict:** ✅ **PASS** — proceed with the `onnxruntime_go` path. No fallback to TFLite or a sidecar is needed.

## What P2.0 asked

> Cross-compile a minimal `onnxruntime_go` test binary for the Brick target, push via adb, and confirm: runtime loads, an openWakeWord model runs on 16 kHz frames, and detection latency/CPU fit budget. If it fails: fall back to a TFLite runtime, or a small wake-word sidecar. No further Phase 2 work proceeds until this passes.

All three confirmations hold.

## Setup

- **Binding:** `github.com/yalue/onnxruntime_go` v1.31.0 (CGo wrapper that `dlopen`s `libonnxruntime.so` at runtime — no ORT headers needed at build time).
- **Runtime:** Microsoft prebuilt `onnxruntime-linux-aarch64-1.26.0` (`libonnxruntime.so`, ~19.5 MB).
- **Models:** stock openWakeWord v0.5.1 ONNX pipeline — `melspectrogram.onnx` (1.1 MB) → `embedding_model.onnx` (1.3 MB, Google speech-embedding) → `hey_jarvis_v0.1.onnx` classifier (1.3 MB). "Hey BMO" training is P2.2; a stock wake word is sufficient to prove feasibility.
- **Cross-compile:** `CGO_ENABLED=1 GOOS=linux GOARCH=arm64` inside `bmo-pak-tg5040-dev` (the existing LoveRetro toolchain image; `CC=aarch64-nextui-linux-gnu-gcc`). Same path `release.sh` already uses.
- **Device:** Brick (tg5040, serial `4c00…2229d`), aarch64, 4 cores, ~998 MB RAM. Governor `schedutil` running at 1.8 GHz (max 2.0 GHz) during the run.
- **Spike code:** `cmd/wakeword-spike/` (throwaway; not wired into bmo-pak). Streams 80 ms / 1280-sample frames through the pipeline mirroring openWakeWord's structure (mel ring → embedding every 8 mel frames → classifier on last 16 embeddings) and reports per-model latency, realtime factor, and process RSS.

## Results (on device)

Per-model latency, steady state, by intra-op thread count:

| threads | melspec avg | embedding avg | embedding p95 | classifier avg | RSS |
|--------:|------------:|--------------:|--------------:|---------------:|----:|
| 1 | 3.4 ms | 27.6 ms | 40.8 ms | 1.1 ms | 41.3 MiB |
| **2** | ~3 ms | **15.4 ms** | **16.5 ms** | ~1 ms | 40.2 MiB |
| 4 | — | 56.4 ms ⚠️ | 93.2 ms | — | 40.0 MiB |

**Embedding dominates** (~90% of compute). It scales well 1→2 threads (27.6→15.4 ms) but **regresses badly at 4 threads** (56 ms) — oversubscribing all cores contends with system processes and the model's own parallelism overhead. **Use 2 intra-op threads.**

For comparison, on the x86 dev host the full step ran ~0.7 ms (111× realtime, 40.7 MiB).

### Honest per-step cost

The spike's headline "FULL STEP avg" (21.8 ms @ 1 thread) is **optimistic**: feeding `melspectrogram` isolated 1280-sample chunks yields only ~5 mel frames/step (window-edge loss), so embedding fired ~0.6×/step. A **correctly stream-aligned** detector feeds continuous audio → 8 mel frames/step → the embedding (and classifier) run **every** step. The realistic steady-state per-frame cost is therefore the sum of all three models:

- **1 thread:** ≈ 3.4 + 27.6 + 1.1 ≈ **32 ms / 80 ms frame** (~40% of one core; ~2.5× realtime headroom)
- **2 threads:** ≈ 3 + 15.4 + 1 ≈ **~19 ms / 80 ms frame** (~24% over two cores; ~4× headroom)

Both are comfortably inside the 80 ms frame budget, including p95.

## Budget assessment

| Dimension | Result | Budget | Verdict |
|---|---|---|---|
| Runtime loads on arm64 | Yes; `libstdc++.so.6.0.28` on device is sufficient (no GLIBCXX gap) | must load | ✅ |
| Per-frame latency | ~19 ms (2 thr) / ~32 ms (1 thr), p95 well under | < 80 ms | ✅ ~4×/2.5× headroom |
| Continuous CPU while detecting | ~16% of one core (2 thr) / ~27% (1 thr) | detector is "tiny", idle-only | ✅ |
| **RAM (RSS)** | **~40–41 MiB, stable across all runs, no growth** | the known OOM risk; ~500 MB free | ✅ huge margin |
| Binary/asset size | ORT `.so` ~19.5 MB + models ~3.7 MB | pak asset | ⚠️ note below |

## Notes & implications for the rest of Phase 2

- **Threads:** default the detector to **2 intra-op threads**; never 4. Revisit if the detector shares cores with active STT/TTS bursts.
- **CPU/battery:** ~16–27% of one core *continuously* while the detector is on is non-trivial for an always-on mic. The spec's mitigations hold up: detector runs **only while idle and wake-word-enabled**, and the `performance` governor is requested **only during STT/TTS bursts** (P2.7), not for the detector.
- **Asset shipping (open question in spec):** the ORT `.so` (~19.5 MB) dwarfs the binary and models. Recommend shipping it as a **pak asset / `lib/` bundle** loaded via `SetSharedLibraryPath`, not embedded — keeps the bmo-pak binary lean and mirrors how `libSDL2` is already bundled by `release.sh`.
- **Stream alignment (for P2.2):** the real detector must buffer raw audio and feed `melspectrogram` continuously (8 frames/step), not per-isolated-chunk, or scores will be wrong. openWakeWord's normalization (`mel/10 + 2`) is required before the embedding model.
- **Latency to detection:** classifier sees 16 embeddings ≈ 1.28 s of context; end-to-end wake latency is dominated by that window, not by compute.

## Reproduce

```bash
# artifacts live in spike-artifacts/ (gitignored): ORT .so (x64 + aarch64) + openWakeWord onnx
# host sanity:
CGO_ENABLED=1 go build -o /tmp/wakeword-spike-x64 ./cmd/wakeword-spike
/tmp/wakeword-spike-x64 -lib spike-artifacts/onnxruntime-linux-x64-1.26.0/lib/libonnxruntime.so \
  -models spike-artifacts/onnx -seconds 20 -threads 2

# arm64 cross-compile (in toolchain container) + on-device:
#   docker run ... bmo-pak-tg5040-dev: CGO_ENABLED=1 GOOS=linux GOARCH=arm64 go build -tags netgo ...
#   adb push binary + libonnxruntime.so (aarch64) + onnx/  →  run with -lib/-models
```
